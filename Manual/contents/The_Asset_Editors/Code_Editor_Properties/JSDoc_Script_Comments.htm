Commentaires sur le script JSDoc

Si vous souhaitez que vos fonctions personnalisées soient complétées par du code et que les arguments requis soient affichés de manière spécifique dans l'éditeur de code, vous devez ajouter des commentaires de style JSDoc. Ces commentaires sont utilisés pour indiquer à la fonction de complétion automatique comment la fonction doit être utilisée et remplie dans l'éditeur de script.

Le format d'un en-tête de fonction typique est le suivant : le nom de la fonction, sa description, puis la liste des différents arguments (paramètres) que prend la fonction, en veillant à commencer chaque ligne par un triple backslash "///", qui indique à GameMaker d'analyser le commentaire comme étant du style JSDoc. Vous pouvez également envelopper vos commentaires JSDoc dans /**... */ en veillant à ce que /** soit sur sa propre ligne.

Les commentaires eux-mêmes doivent recevoir un identifiant (précédé de "@") et un contenu. Les identifiants disponibles sont les suivants :

Identifiant Contenu Syntaxe
@function / @func Le nom complet de la fonction, y compris les arguments @func my_func(args)
@description / @desc Une description générale de ce que fait la fonction @desc <Description goes>
@param / @parameter / @arg / @argument Informations sur les arguments, avec une option type @param {type} name <Parameter description>
@return / @returns Type de données que la fonction renvoie @return {type} <Return description>
Plume uniquement
@url URL for the function, opens when you press F1 on the function name or use the RMB -> Go To Declaration option. @url https://...
@pure Marque la fonction comme pure @pure
@ignore Masque la fonction de l'autocomplétion de Feather. @ignore
@deprecated Marque la fonction comme dépréciée @deprecated
@context / @self Définit le contexte de la fonction, que Feather auto-complete utilise pour fournir des informations contextuelles. Peut être un objet ou un constructeur. @self <object>
Legacy Code Editor only
@function / @func The full function name including arguments @func my_func(args)

Si vous n'avez pas spécifié de type pour vos arguments ou votre valeur de retour, et que vous utilisez Feather, il supposera automatiquement un type de données pour eux, sur la base du corps de votre fonction.

Voir les informations sur les types de données pour @param et @return sur Feather Data Types.

Exemples

Pour avoir une idée de la façon dont cela fonctionne lorsque vous écrivez vos propres fonctions, prenons cet exemple de base :

function is_same_object(_id, _obj)
{
    if (_id.object_index == _obj)
    {
        return true;
    }
    else return false;
}

Tout ce que fait ce script est de vérifier si une instance a la même object_index qu'un objet donné et serait appelé simplement comme :

if is_same_object(id, obj_Player)
{
    instance_destroy()
}

Cependant, si vous écrivez cela dans l'éditeur de code, vous verrez directement les noms des variables d'argument (_id et _obj), ce qui, dans la plupart des cas, n'est pas très descriptif. Vous pouvez utiliser JSDoc pour définir des noms et des descriptions d'arguments personnalisés, ainsi que des informations sur la fonction :

/// @function                is_same_object(inst_id, object_id)
/// @description             Check if the given instance belongs to the given object.
/// @param {Id.Instance}     inst_id    The unique instance ID value of the instance to check.
/// @param {Asset.GMObject}  object_id  The object index to be checked against.
/// @return {Bool}

function is_same_object(_inst_id, _object_id)
{
    return _inst_id.object_index == _object_id;
}

Vous pouvez entourer le nom d'un argument entre [] crochets pour indiquer qu'il est facultatif. L'éditeur de code attendra alors un nombre quelconque d'arguments entre le nombre minimum d'arguments requis et le nombre total d'arguments.

Par exemple, voir la fonction suivante :

/// @function    animate_position(end_x, end_y, start_x, start_y)
/// @desc        Animates the instance to ending point, from optional starting point
/// @arg end_x
/// @arg end_y
/// @arg [start_x]
/// @arg [start_y]

function animate_position (x1, y1, x2, y2)
{
    // Function code
}

Les arguments start_x et start_y sont marqués comme facultatifs, ce qui signifie que l'éditeur de code s'attend maintenant à 2 à 4 arguments, comme on peut le voir dans le message d'avertissement :
