Options de jeu sur Android

Cette section présente les différentes options disponibles qui contrôlent la façon dont vos projets de jeu Android seront compilés. Les différentes sections sont les suivantes :

Général

La première partie de la section Général est consacrée aux informations sur le produit et aux paramètres de construction. Vous devez remplir les informations sur le produit, qui comprennent le nom d'affichage du projet, les informations sur le paquet, qui consistent en un domaine, une société et un produit. Les informations sur le paquet ne peuvent être constituées que de lettres de A à Z et de chiffres de 0 à 9, sans symboles ni autres caractères spéciaux, et ne doivent pas comporter de majuscules. Ces informations seront ensuite utilisées pour générer un ID de paquet au format "com.company.myappname".

Vous devez ensuite définir les outils de construction ainsi que les versions Target, Minimum et Compile SDK.

REMARQUE: il existe également une option pour la version "Support Library" à utiliser. À partir de la version 2.2.4 de GameMaker, cette option n'est plus nécessaire et toute valeur utilisée ici sera ignorée. Ce paramètre est conservé pour les anciens projets qui nécessitent une version antérieure à la 2.2.4 sur runtime.

Les outils de construction (et la bibliothèque de support, le cas échéant) sont utilisés lorsque GameMaker construit votre jeu pour la cible Android et doivent être réglés sur les valeurs appropriées en fonction des outils que vous avez installés à l'aide de Android Studio.

  • Le SDK cible indique que vous avez testé votre application sur (jusqu'à et y compris) la version que vous spécifiez ici. Il s'agit simplement de donner au système d'exploitation Android une idée de la manière dont il doit traiter votre application en termes de fonctionnalités du système d'exploitation. À toutes fins utiles, la plupart des applications voudront définir le SDK cible sur la dernière version publiée de l'API (telle qu'utilisée par le paramètre Compile SDK). Cela permettra à votre application de s'afficher aussi bien que possible sur les appareils Android les plus récents. Notez qu'à partir de novembre 2019, Google a fait du niveau 28 de l'API la cible requise.
  • La version minimale du SDK est le niveau d'API minimum qui permettra de faire fonctionner votre projet. À partir de novembre 2019, Google a fait du niveau d'API 16 le minimum absolu requis, mais si vous ajoutez des extensions, alors elles peuvent nécessiter un niveau d'API minimum plus élevé.
  • La version Compile SDK est la version de l'API avec laquelle le projet est compilé. Cela signifie que vous pouvez utiliser les fonctionnalités de l'API Android incluses dans cette version de l'API. Si vous essayez d'utiliser les fonctionnalités de l'API 16 - par exemple - mais que vous définissez le Compile SDK 15, vous obtiendrez une erreur de compilation. Toutefois, si vous définissez le SDK de compilation sur 28, vous pourrez toujours exécuter l'application sur un appareil de l'API 26 (ainsi que sur toutes les autres versions précédentes).

Vous pouvez choisir l'un des paramètres cibles prédéfinis en cliquant sur le bouton " Pre-populate SDK values to chosen API level", qui ouvrira une liste de différentes API à choisir. En sélectionnant l'une d'entre elles, tous les fichiers SDK seront remplis de valeurs appropriées que vous pourrez ensuite utiliser ou modifier selon vos besoins.

REMARQUE: vous devez avoir installé les API appropriées dans le SDK Manager de Android ou via Android Studio pour que vos projets se compilent correctement.

La deuxième partie de la section Général couvre les architectures de CPU pour lesquelles construire, avec les options suivantes disponibles (vous pouvez les sélectionner toutes ou seulement celles dont vous avez besoin, mais notez que chacune d'entre elles ajoutera à la taille finale de l'apk compilé) :

  • Construire pour ARMV5 (coché par défaut)
  • Construire pour ARMV7 (coché par défaut)
  • Construire pour x86 (coché par défaut)
  • Construire pour Mips (coché par défaut)
  • Construction pour ARM64
  • Construction pour x86_64

Après l'avoir configuré, vous pouvez sélectionner les différentes orientations sur lesquelles votre jeu peut être exécuté pour une expérience utilisateur optimale. Les options disponibles sont les suivantes :

  • Portrait
  • Portrait-flippé
  • Paysage
  • Retournement du paysage

Enfin, vous disposez des options diverses suivantes :

  • Activez le support Bluetooth/iCade: Ceci ajoutera le support pour les manettes de jeu Bluetooth ou iCade à votre projet de jeu. Cette option est activée par défaut.
  • Exécuter l'analyse de code Lint: Lorsque cette option est activée, les outils de compilation exécutent une analyse de code Lint sur le projet lors de la compilation, ce qui permet une vérification plus approfondie des erreurs et l'affichage de plus d'informations dans la fenêtre de sortie au fur et à mesure de la compilation du projet (un fichier LOG sera également créé, avec le chemin d'accès au fichier affiché à la fin de la compilation dans la fenêtre de sortie). Notez que tout ce qui est signalé par l'analyse lint comme une erreur n'est pas nécessairement problématique pour votre jeu, et avoir cette option activée peut signifier que votre jeu ne compile plus, même si lorsqu'elle est désactivée, il compile et fonctionne bien. Essentiellement, cela ne concerne que debugging, et l'option est désactivée par défaut.
  • Emplacement d'installation: Vous pouvez définir ici l'emplacement d'installation par défaut de votre jeu. Il peut être réglé sur Automatique, auquel cas le jeu sera installé à l'emplacement choisi par l'utilisateur sur son appareil, ou sur Préférer l'extérieur, qui essaiera toujours d'installer le jeu sur le stockage externe de l'appareil.

Vous pouvez également définir la marge de veille d'Android ici. Cette option est liée à la réduction du bégaiement lors de l'exécution de votre jeu sur des systèmes spécifiques. En fait, si votre jeu tourne plus vite que la vitesse de votre pièce, GameMaker va "dormir" pendant le temps restant, mais cette mise en veille peut être assez imprécise et vous pouvez souvent dormir plus longtemps que nécessaire, ce qui fait que votre image prend plus de temps et provoque des bégaiements. Pour contourner ce problème, nous pouvons dormir moins longtemps, puis nous asseoir dans une boucle serrée pour le reste du temps afin de le rendre plus précis - bien que le problème avec le fait de s'asseoir dans une boucle est qu'il entraîne une augmentation de l'utilisation du CPU, ce qui à son tour peut entraîner une augmentation de la température du CPU et une utilisation accrue de la batterie de votre appareil. Par défaut, cette valeur est réglée sur 4, et dans 99,99% des cas, cela vous conviendra, mais pour les appareils bas de gamme ou pour les appareils avec beaucoup de processus d'arrière-plan en cours d'exécution, cela peut ne pas être une solution idéale et une valeur de 5 à 10 peut être nécessaire. Notez cependant qu'il s'agit d'une configuration spécifique à un appareil et que ce qui fonctionne sur votre appareil peut ne pas convenir à un autre utilisateur. En cas de doute, laissez la valeur 4.

Icônes adaptatives

Android 8.0 (niveau 26 de l'API) a introduit des icônes de lanceur adaptatives pour vos jeux, et ces icônes peuvent afficher une variété de formes sur différents modèles d'appareils. Pour y remédier, vous devez fournir un certain nombre d'images d'avant-plan et d'arrière-plan pour que vos icônes s'affichent à des tailles différentes. Ces tailles d'icônes sont les suivantes :

  • 81x81 pixels (LDPI)
  • 108x108 pixels MDPI)
  • 162x162 pixels (HDPI)
  • 216x216 pixels (XHDPI)
  • 324x324 pixels (XXHDPI)
  • 432x432 pixels (XXXHDPI)

Vous pouvez fournir chaque image d'arrière-plan et de premier plan individuellement ou choisir de générer les icônes à partir d'un fichier. Cela signifie que GameMaker générera les images d'icônes requises à partir d'un fichier de base (XXXHDPI). Vous devez fournir une image pour le premier plan et une autre pour l'arrière-plan, puis cliquer sur le bouton "Générer" pour que les icônes soient créées pour vous. Les images doivent toutes être au format *.png.

Il est important de comprendre que vos icônes adaptatives peuvent être masquées sur un appareil donné. Vous avez donc également la possibilité de voir comment elles se présenteront avec ces masques dans le menu Aperçu des icônes adaptatives:

Notez qu'il s'agit simplement d'un aperçu de l'apparence de l'icône avec le masque sélectionné, et que le paramètre n'affectera pas la façon dont les icônes sont exportées et affichées, car cela dépend de l'appareil et du système d'exploitation sur lesquels le jeu sera exécuté. Notez également que ces icônes ne sont pas créées pour vous dans le cadre de la génération d'images réalisée à l'aide de l'outil Project Image Generator.

Graphiques

Ici, vous pouvez modifier les détails suivants relatifs à la façon dont votre jeu sera affiché, les options suivantes étant disponibles :

  • Interpoler les couleurs entre les pixels: Active l'interpolation linéaire, qui a pour effet de "lisser" les pixels. Pour des graphiques de pixels nets, il est préférable de désactiver cette fonction, mais si vous avez de beaux mélanges alpha et des graphiques aux bords lissés, il est préférable de l'activer. Cette option est activée par défaut.
  • Profondeur de couleur de l'écran: Ceci peut être utilisé pour définir la profondeur de couleur pour le rendu à 16 ou 24 bits. Cela affectera la compatibilité avec certains appareils plus anciens s'il est réglé sur 24 bits, et augmentera également l'ashmem qui est nécessaire (c'est la mémoire partagée sur les appareils AmazonFire).
  • Prise en charge des périphériques: Cette option permet de déterminer s'il faut prendre en charge uniquement les appareils dotés d'un GPU dédié, ou tous les appareils. En fait, si votre jeu comporte des fonctions de dessin avancées, comme le mélange alpha, le mélange additif, les surfaces, etc., vous devez le limiter aux appareils dotés d'un GPU, sinon vous risquez de recevoir des commentaires négatifs et une mauvaise évaluation de votre jeu en raison d'erreurs graphiques ou de plantages sur les appareils plus anciens.
  • Mise à l'échelle: Ici, vous pouvez choisir de mettre à l'échelle le canevas de dessin en conservant le rapport d'aspect de l'écran de l'appareil - en ajoutant un "rembourrage" autour des bords pour s'adapter à l'écran - ou vous pouvez choisir d'étirer le canevas de dessin pour l'adapter à l'écran de l'appareil.

Enfin, il est possible de définir la taille de la page de texture. La taille par défaut (et la plus compatible) est de 2048x2048, mais vous pouvez choisir entre 256x256 et 8192x8192. Il y a aussi un bouton " Preview" qui génère les pages de texture pour cette plate-forme et ouvre une fenêtre pour que vous puissiez voir à quoi elles ressemblent. Cela peut être très utile si vous souhaitez voir comment les pages de texture sont structurées et éviter d'avoir des pages de texture plus grandes (ou plus petites) que nécessaire. Pour plus d'informations sur les pages de texture, veuillez cliquer ici.

NOTE: Sachez que plus la taille de la page de texture est grande, moins votre jeu sera compatible avec les différents navigateurs et appareils.

Icônes

Cette section vous permet d'ajouter les diverses images d'icônes dont votre projet aura besoin pour les différentes pages de la boutique et les différents appareils. Toutes les images d'icônes doivent être créées sous forme de fichiers 24 bits *.png aux tailles appropriées indiquées.

Il convient de noter que GameMaker dispose d'un outil de génération d'images de projet qui peut être utilisé pour créer automatiquement toutes les images requises pour toutes les différentes plates-formes cibles pour lesquelles votre jeu est compilé. Si vous utilisez cet outil, vous devez réviser les images créées pour vous assurer qu'elles correspondent à vos besoins. L'outil génère aussi bien des icônes que des images (voir la section ci-dessous).

Images

Vous pouvez définir ici la couleur de bordure qui sera utilisée comme arrière-plan pour les écrans d'accueil qui ne sont pas configurés pour remplir tout l'écran. Vous pouvez également ajouter des écrans d'accueil à votre fichier de jeu final, pour les modes paysage et portrait , qui s'afficheront pendant le chargement du jeu sur l'appareil. Cet écran doit être un fichier 24bit *.png, et il est recommandé qu'il soit de la même taille que la première pièce (ou vue) de votre jeu. Si vous souhaitez que l'écran d'accueil s'affiche pendant une durée spécifique, vous pouvez également le définir ici entre 0 et 10 secondes (la durée par défaut de 0 signifie qu'il ne s'affichera que pendant le chargement de la ressource). Vous pouvez également définir le mode d'affichage de l'écran de démarrage à partir des options suivantes :

  • Conserver le rapport d'aspect: Cette option permet de conserver le rapport d'aspect de l'image par rapport à l'écran de l'appareil, en ajoutant un "rembourrage" autour des bords si nécessaire.
  • Recadrer pour adapter: Cette option permet de conserver l'image à l'échelle de l'écran de l'appareil et de recadrer (ou de rembourrer) les bords si nécessaire.
  • Étirer: Cette option permet d'étirer l'image pour qu'elle remplisse l'écran de l'appareil (et peut déformer l'image affichée).

Enfin, vous pouvez définir l'image de la bannière pour le jeu sur les appareils TV Android, qui est également une image 24bit *.png.

Emballage

L'une des caractéristiques de l'utilisation de Google Play pour vos jeux Android est la possibilité d'accorder une licence à vos jeux. Cela signifie que vos jeux seront contrôlés par Google de sorte que seules les personnes qui les auront téléchargés sur leur marché pourront y jouer, ce qui évitera la distribution de copies illégales de votre produit.

Pour activer cette fonction dans votre jeu, vous devez copier votre clé publique dans la case intitulée " Licence Google" Public Key. Vous pouvez trouver cette licence en vous rendant sur votre page de développeurs Google Play, puis en sélectionnant le jeu que vous souhaitez modifier. Une fois sur la page du profil du jeu, faites défiler la page jusqu'à la section intitulée "Services & APIs". Vous y trouverez la clé Public nécessaire pour activer cette fonctionnalité.

Une fois activée, si une personne qui n'est pas autorisée à posséder votre jeu tente d'y jouer, elle verra apparaître une boîte de message contextuel qui l'informera que la licence est introuvable sur son appareil, et lui donnera la possibilité de réessayer et de tester à nouveau, ou de quitter le jeu.

Une autre caractéristique de la licence est que votre jeu peut utiliser des extensions APK. Il s'agit d'un moyen de contourner la taille maximale des applications de 50 Mo imposée par Google Play (et uniquement Google Play). Il créera votre jeu en deux parties : un petit fichier *.apk et un plus grand fichier *.zip qui contiendra tous les actifs de votre jeu, tous deux étant téléchargés sur le magasin. Lorsque l'utilisateur lance votre jeu pour la première fois, le fichier zip est téléchargé et décompressé. Vous trouverez plus de détails sur cette option ici.

Vous pouvez également utiliser Google Dynamic Asset Delivery pour créer des paquets d'applications dont la taille peut atteindre 1 Go en utilisant l'option d'exportation *.aab. Cela permet de regrouper toutes les ressources du jeu en tant que ressources d'installation qui sont installées en même temps que l'application elle-même. Vous trouverez plus de détails sur cette fonctionnalité ici.

Enfin, vous avez la possibilité de préparer le package final de l'application comme un jeu Android TV (vous ne devez cocher cette case que si votre jeu est conforme aux directives Android TV).

Permissions

Dans cet onglet, vous pouvez modifier certaines des permissions que votre jeu peut demander dans le manifeste Android. GameMaker attribuera automatiquement les autorisations requises en fonction des fonctions utilisées par le jeu. Cependant, il se peut que vous ayez besoin d'une autorisation qui n'a pas été correctement attribuée (par exemple, lorsque vous utilisez des extensions personnalisées), auquel cas cochez la case de l'autorisation requise.

La liste ci-dessous présente les autorisations disponibles :

  • WRITE_EXTERNAL_STORAGE: Permet à votre jeu d'écrire sur le stockage externe.
  • READ_PHONE_STATE: Autoriser la lecture seule access à l'état du téléphone.
  • ACCESS_NETWORK_STATE: Permettre à votre jeu de access informations sur les réseaux.
  • INTERNET: Permettre à votre jeu d'ouvrir des sockets réseau.
  • BLUETOOTH: Permet à votre jeu de se connecter à des appareils Bluetooth jumelés.
  • RECORD_AUDIO: Permet à votre jeu d'enregistrer l'entrée audio du microphone.

REMARQUE: si vous n'êtes pas sûr d'avoir besoin de ces éléments, vous devriez probablement les laisser décochés par défaut et laisser GameMaker s'occuper des autorisations pour votre jeu.

La section Permissions de Android comporte également une zone dans laquelle vous pouvez injecter des permissions dans les balises d'application Android de Androidmanifest.xml. Vous trouverez une liste complète des balises disponibles dans la documentation de Android ici :

  • Développeur Android : Tags d'application

Il convient de noter que pour l'API cible 28 et les versions ultérieures, vous ne serez plus autorisé à utiliser des appels Web externes à l'aide de "http" - "https" est requis à la place - à moins que vous n'ajoutiez la balise suivante :

android:usesCleartextTraffic="true"

Social

Cette section permet d'activer différentes fonctions sociales dans votre jeu Android. Les options suivantes peuvent être choisies :

  • Google - Activer Cloud Saving: Si vous cochez cette case, Google Cloud Saving sera activé et vous serez invité à télécharger et à installer l'extension correspondante depuis le Marketplace. Voir la section sur le Cloud Saving de la page Fonctions asynchrones.
  • Google - ID des services Google: Cette option vous permet de saisir l'ID des services Google de votre application.
